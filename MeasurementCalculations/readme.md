- Ich kenne das MVVM Design Pattern. 
MVVM steht fuer Model-View-ViewModel und besteht aus drei Hauptkomponenten:
1. Model: Das Model repraesentiert die Daten und Geschaeftslogik der Anwendung. 
2. View: Die View ist die Benutzeroberflaeche, die dem Benutzer praesentiert wird. 
3. ViewModel: Das ViewModel ist der Vermittler zwischen Model und View. Es enthaelt die Logik, um die Daten aus dem Model aufzubereiten und an die View zu binden.
Es ist ein Software-Architekturmuster, das die Trennung der drei Hauptkomponenten foerdert und dafuer sorgt, dass die Anwendung testbar ist. 

- Das Code Behind Pattern wird fuer kleine Projekte verwendet. Dabei ist die Logik im Code Behind der View enthalten. Die Wartbarkeit und Testbarkeit ist eingeschraenkt.

- Es gibt das IOC Pattern, Inversion of Control. Bei dem kann man z.B. verschiedene Dienste erstellen und diese dann in der Klasse (ueber Konstruktorueberladung) verwenden. Die Dienste werden in einem IOC-Container/ViewModelLocator gespeichert.
